**Date:** 2026-07-03

**Author:** Claude Code (Session — "LANE 3 — On-Site Personalization")

## What Was Built

The full on-site personalization slice against the frozen `OnSiteElement` (POPUP /
STICKY_BAR / EMBED) and `AbTest` (entityType `ONSITE_ELEMENT`) tables — merchant
config, segment-based delivery, deterministic A/B variants, a public delivery
endpoint the storefront SDK calls, and SDK rendering with impression/conversion
telemetry. No schema was touched; the lane applied the frozen migrations with
`db:migrate:deploy` only.

**Shared contract (interface-first).** A `// lane:onsite` block appended to
`packages/shared/src/types.ts` + barrel: `OnSiteElementConfig`,
`OnSiteDisplayRules` (trigger + params + frequency), `OnSiteVariant`,
`OnSiteDeliveryRequest`/`Response`, and the `onsite_impression` /
`onsite_conversion` event-name constants. This shapes the two Json columns on
`OnSiteElement` (`config`, `displayRules`) and the `AbTest.variants` Json, and is
the single source of truth for the API↔SDK↔web boundary.

**API — services (`apps/api/src/services/onsite/`).**
- `targeting.service.ts`: pure, unit-tested helpers — `hashToBucket` (FNV-1a over
  `anonId:abTestId` → 0..99, stable per visitor), `pickVariant` (allocation-window
  walk with under-sum fallback), `matchesPage`, `resolveElementConfig`
  (RUNNING → deterministic variant; WINNER_DECIDED → winner to everyone; none →
  base). Plus `resolveVisitorSegments` (tenant-scoped: known customer by id or by
  `anonIds` containment → active memberships; anonymous → empty set) and
  `selectElementsForVisitor` (ACTIVE elements, segment gate, page-pattern gate,
  A/B resolution, priority order).
- `stats.service.ts`: `getElementStats` recomputes impression/conversion counts
  (+ per-variant) from `engageiq.events` via the exported ClickHouse client
  (sanctioned escape hatch — never imports `@clickhouse/client`). Fully
  CH-independent: returns a zeroed result with `available:false` when CH is down.

**API — routes (`apps/api/src/routes/onsite/`).** `schema/service/controller/index`.
Authenticated CRUD (`POST/GET/PUT/DELETE /api/v1/onsite`), A/B lifecycle
(`POST /:id/ab-test`, `/:id/ab-test/:testId/stop`, `/:id/ab-test/:testId/decide`),
`GET /:id/stats`, and a **public unauthenticated** `POST /api/v1/onsite/deliver`
(rate-limited, merchant verified, empty set for unknown merchant to avoid
fingerprinting). The plugin registers the public route in the outer scope and the
CRUD routes in an encapsulated child scope carrying the `authenticate` hook, so
one `app.register` line covers both. Every query is `merchantId`-scoped.

**SDK (`packages/sdk/`).** New self-contained `onsite.ts` renderer wired into the
existing `init()`. On load it gathers cart value (`/cart.js`), current + previously
viewed products (localStorage), POSTs `/api/v1/onsite/deliver`, and for each
returned element enforces the trigger client-side (new-visitor / exit-intent /
timed / cart-value / product-view-restock), page-pattern, and frequency
(session/day/ever) gating. Renders popup (overlay card), sticky bar (fixed bar), or
inline embed (injected at a CSS selector), monochrome inline styles, with
personalization-token substitution and email capture (→ existing `/v1/sdk/identify`).
Impressions and conversions flow through the **same** `track` → `/v1/sdk/events`
→ ClickHouse pipeline (no parallel path). Bundle: **4.68 KB gzipped** (under the
5 KB target).

**Web (`apps/web/app/routes/on-site.*` + `components/onsite/*`).** List, create, and
detail/edit routes built entirely from `~/components/ui` (strict monochrome — no
hue). Detail page shows performance StatCards (graceful "—" when analytics
unavailable), the edit form, and an A/B panel (start a 2-variant test, per-variant
stats, roll out a winner, stop). Flipped the shell `nav.ts` On-Site entry off
`soon` (the one sanctioned shared edit, per DESIGN_SYSTEM.md §5).

## Decisions Made This Session (all pre-approved by Abdullah)
1. **Public delivery endpoint** unauthenticated + rate-limited, keyed by merchant +
   visitor context (storefronts have no auth).
2. **Impressions/conversions reuse the existing SDK→ClickHouse pipeline** (new event
   types), counts recomputed from ClickHouse on read — no parallel ingestion.
3. **Deterministic A/B bucketing** via FNV-1a hash of `anonId:abTestId` so a visitor
   stays in the same variant. WINNER_DECIDED rolls the winner out to everyone.
4. Server decides element *eligibility* (segment + page); the SDK enforces trigger
   *timing* client-side (it holds the exit-intent/timer/cart/product signals).
5. ClickHouse `CLICKHOUSE_DATABASE` set back to `engageiq` (the events table is
   hardcoded `engageiq.events`; a per-lane CH DB would be unused).

## Known Issues Left Open
- **ClickHouse was down locally**, so live impression/conversion *stats* weren't
  observed end-to-end. The stats path is CH-independent (verified `available:false`
  degradation) and the write path reuses the already-proven `/v1/sdk/events`
  pipeline; a live-CH check is a browser/integration follow-up.
- `impression_count` / `conversion_count` columns on `OnSiteElement` remain the
  frozen PLACEHOLDER zeros — stats are computed from ClickHouse on read rather than
  denormalized back onto the row (no writer wired; add a small precompute worker if
  denormalized counts are wanted on the list view).
- `product_view_restock` "previously viewed" is enforced client-side from SDK
  localStorage; the server accepts `viewedProductIds` but currently gates only on
  segment + page (the trigger itself is client-side). Fine for v1.
- No dedicated E2E/browser test (per project norms — Abdullah verifies in browser).
