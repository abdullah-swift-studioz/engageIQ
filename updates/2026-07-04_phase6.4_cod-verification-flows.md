## What Was Built

The automated COD verification flow: when the (already-merged) fake-order gate flags a COD order
`verificationStatus=PENDING_VERIFICATION`, this lane contacts the customer to confirm intent before
the order ships, escalates across channels on no-response, and auto-cancels on final timeout. It
integrates the merged two-way WhatsApp conversation engine (consumes its verification hand-off seam)
and the merged channel-dispatch seam.

**Flow (all timers configurable per merchant):**
- **Decoupled entry** — a repeatable `scan` job (every 60s) picks up `PENDING_VERIFICATION` COD orders
  that have no `VerificationAttempt` yet and enqueues a `start`. The fake-order gate is never touched;
  we poll for its output, so there is zero coupling to gate internals.
- **Escalation ladder** — `start` = attempt #1; `reminder` jobs = attempts #2..N; `timeout` = finalize.
  Default ladder (guide §7.4): **15min WhatsApp → 2h WhatsApp reminder → 4h SMS**, then **auto-cancel at
  6h**. Each tick is a delayed BullMQ job with a deterministic `jobId` (idempotent).
- **Two-way reply (WhatsApp)** — each WhatsApp attempt sends the prompt via `dispatchChannel` AND arms a
  `WhatsAppConversation` (`contextType='verification'`, `contextId=<codOrderId>`) via the conversation
  engine's `reuseOrArmConversation` seam. The customer's YES/NO reply flows through the engine's
  `resolveVerificationReply`, which now calls our `applyVerificationDecision` (the seam the wa-conversation
  lane left as a `console.info`). CONFIRM → order **VERIFIED** + status **CONFIRMED** (released). CANCEL →
  **AUTO_CANCELLED** + status **CANCELLED**.
- **SMS / IVR** — SMS is a one-way reminder via `dispatchChannel` (no SMS inbound engine in scope). IVR is
  a Fixerr adapter reading env creds; **no creds → graceful mock/skip** (never fabricates a decision).
- **Finalize** — on the auto-cancel deadline with no confirm: mark awaiting attempts `NO_RESPONSE`, then
  either **AUTO_CANCELLED** (default) or leave `PENDING_VERIFICATION` for **manual review** (`autoCancel:false`).
- **Idempotency / races** — the ORDER's `verificationStatus` is the single source of truth: every job and
  decision no-ops once it leaves `PENDING_VERIFICATION`, so a reply landing the same instant a timer fires
  can never double-decide. Verified live with a real reply-vs-timer scenario and unit tests.
- **Analytics + dashboard** — `GET /api/v1/verifications/stats` (confirm / cancel / no-response / response
  rates, revenue saved = Σ auto-cancelled amounts, per-channel breakdown). Web pages `/verifications`
  (queue + KPI tiles + per-channel table) and `/verifications/:id` (order facts + attempt timeline +
  manual confirm/cancel/re-send). Both built entirely from the monochrome design system (state by
  shade/border/icon, never hue).

**Enum note:** `CodVerificationStatus` has no plain `CANCELLED` — both a customer cancel and a no-response
timeout map the ORDER to `AUTO_CANCELLED`. The nuance (customer NO vs never replied) is preserved at
`VerificationAttempt.status` (`CANCELLED` vs `NO_RESPONSE`) and surfaced in analytics.

## Decisions Made This Session
- **Adopted the frozen `VerificationJob` contract** (`start{channel}` / `reminder{attemptNumber}` / `timeout`) from schema-freeze-v2 rather than inventing a job shape. `start` = attempt #1, `reminder` = #2..N, `timeout` = finalize. Added only a separate `CodVerificationScanJob` for the sweep.
- **This lane owns ALL flow timing** via the `cod-verification` queue (not the conversation-timeout queue, which is WhatsApp-only) because SMS/IVR escalation must be driven too. Arming a conversation sets `awaitingReplyUntil` for the WhatsApp reply window; the escalation transitions are driven by our own delayed jobs. No conversation-timeout jobs are scheduled by this lane, so the engine's `verificationTimeout` seam is intentionally left unused.
- **Order `verificationStatus` is the single race guard** — every job/decision is idempotent against it. Reply-vs-timer and job retries are benign.
- **IVR never fabricates a decision** — no Fixerr creds → mocked/skipped, order proceeds to auto-cancel.
- **`AUTO_CANCELLED` for both cancel + no-response** at the order level (frozen enum); nuance kept at attempt level.

## Deviations from Roadmap
- Added manual agent actions (`POST /:id/confirm|cancel|start`) beyond the guide's spec — needed to resolve orders "held for manual review" (`autoCancel:false` path) and for operational re-triggering. New routes, append-only registration, no collision.

## Known Issues Left Open
- **SMS/IVR inbound not wired** — SMS replies and IVR DTMF have no inbound webhook in this lane, so those attempts are one-way; a no-response simply escalates/auto-cancels. WhatsApp is the fully two-way channel. A future SMS-inbound / Fixerr-callback route would close this.
- **WhatsApp business-initiated template** — the verification prompt is business-initiated; production should set `MerchantSettings.codVerification.whatsappTemplateId` (an approved Meta template) so it lands outside the 24h window. Free-form body works where a session is open.
- **Scan cadence fixed at 60s** — fine for typical COD volume; make it configurable if a merchant needs faster pickup. Real-time enrollment would require a hook the fake-order gate does not currently expose (out of scope — gate is off-limits).
- **`listConversations`-style N+1** not applicable here (list uses a single `take:1` include per row); fine at dashboard page sizes.
