# Update: RBAC Enforcement + Agency Accounts

**Date:** 2026-07-03

**Author:** Claude Code (Session — "LANE 9 — RBAC Enforcement + Agency Accounts")

## What Was Built

Full role-based access-control enforcement across the API surface, plus agency
parent→child accounts with account switching and cross-client reporting, and the
Settings → Team & Roles dashboard. Builds on the frozen `AgencyAssignment` table,
the existing `Merchant.agencyId` self-relation, and the `ROLE_PERMISSIONS` matrix.

**1. RBAC enforcement (guide §9.4).**
- A permission-guard factory (`services/rbac/permission-guard.ts`): `requirePermissionByMethod({read,write})` (GET→read, mutations→write) and `requirePermission(perm)`. Guards are appended as route-group `onRequest` hooks that run AFTER each group's `authenticate` (so `request.user.role` is set) and do NOT re-authenticate — a pure authorization check. Deny → `403 FORBIDDEN` (or `401` if unauthenticated).
- Applied to every merchant-data route group: campaigns/segments/journeys (read+write split), customers (+ new `customers:write` for merge), whatsapp-templates + messages (campaigns:*), analytics + recommendations (read-only). Verified live: Analyst can GET but is blocked (403) on POST; Marketer can write; Analyst cannot reach Team.
- Added one permission, `customers:write` (Owner/Admin/Marketer/Agency roles; Analyst stays read-only).

**2. Agency accounts (roadmap 8.3).**
- `services/agency/access.service.ts` is the single authority for cross-tenant access: `listAccessibleMerchants`, `assertChildAccess`, `resolveActingMerchant`, `getReportableMerchantIds`. AGENCY_ADMIN → all children of the agency (`Merchant.agencyId === home`); AGENCY_MEMBER → only children granted via `AgencyAssignment`; every other role → home only.
- **Account switching** via a global acting-merchant `preHandler` (`services/agency/acting-merchant.hook.ts`), registered once in `index.ts` before the route groups. It runs after every group's `authenticate` onRequest hook, so `request.user` is populated. Gated: only an agency user sending a verified `x-acting-merchant-id` header re-scopes the tenant — it swaps `request.user.merchantId` to the child so ALL existing tenant-scoped routes transparently operate on the selected client. `authenticate.ts` is untouched. Access denied → 403 at the preHandler.
- Agency route group (`routes/agency/`, `/api/v1/agency`): `GET /context` (switcher data), `POST /switch` (validate), assignment CRUD (`GET/POST/DELETE /assignments`, gated `agency:manage`), `GET /report` (cross-client headline numbers, gated `agency:manage`). Assignment CRUD validates the member belongs to the agency and the child is owned by the agency (tenant safety).

**3. Settings → Team & Roles.**
- `routes/settings/` (`/api/v1/settings`): `GET /roles` (serialized RBAC matrix, any authenticated user), `GET/POST/PATCH/DELETE /team` (gated `users:manage`). Owner-safety in `services/rbac/team.service.ts`: only an OWNER may grant/change the OWNER role; the last active owner can't be demoted/deactivated/deleted; a user can't deactivate/delete themselves.

**4. Web dashboard.**
- `lib/acting-merchant.server.ts` — the single seam: an `apiFetch(request, …)` helper that forwards the dashboard token + the acting-merchant cookie as `x-acting-merchant-id`, plus the cookie definition.
- `components/shell/AgencySwitcher.tsx` — top-bar switcher (renders for agency users only), loads `/api/agency`, switches the active client (cookie → reload).
- `routes/api.agency.tsx` — resource route backing the switcher (context loader + switch action).
- `routes/settings._index.tsx` + `routes/settings.team.tsx` — settings landing + full Team & Roles UI (invite, change role, activate/deactivate, remove; role reference), built entirely from `~/components/ui` (monochrome design system). `nav.ts` "Settings & RBAC" flipped from `soon` to live.

## Decisions Made This Session
- **Global acting-merchant preHandler over JWT re-mint** (user-approved). The core `authenticate` plugin rejects `token.merchantId !== dbUser.merchantId`, so re-minting a child-scoped token would fail. A gated global preHandler re-scopes transparently without touching core auth. Implemented as a `preHandler` (not `onRequest`) because all `onRequest` hooks — including each group's `authenticate` — complete before any `preHandler`, guaranteeing `request.user` is populated.
- **Permission-based guards (not raw role lists)** — map 1:1 to the guide's §9.4 matrix and are extensible; the one gap (customer merge) got a new `customers:write` permission.
- **Method→permission mapping** (GET=read, mutations=write) keeps the per-group append a single line; analytics/recommendations are read-only regardless of verb.
- **Agency structure ops use `request.homeMerchantId`** (set by the preHandler) so listing children / managing assignments always targets the agency container, never a switched-into client. Team management uses the effective `merchantId`, so an agency admin acting as a client manages that client's team.
- **`Message.toPhone` and other frozen columns untouched** — no schema change; the lane consumed the existing freeze-v2 tables only.

## Known Issues Left Open
- **Existing lanes' loaders don't yet forward `x-acting-merchant-id`.** The API re-scope is complete and the switcher/cookie work; but for an agency user's switch to change what the Campaigns/Segments/Analytics *pages* show, each lane's Remix loaders must call the new `apiFetch(request, …)` seam instead of a bare `fetch` with the raw `DEV_TOKEN`. This is a mechanical integration follow-up (documented in `acting-merchant.server.ts`). My own settings/agency pages already use it.
- **Analytics `/realtime` returns 500 on the fresh lane DB** — pre-existing/environmental (ClickHouse realtime query on an events store with no data); confirmed the RBAC guard PASSES the request through (500, not 403), so it is out of this lane. Analytics lane / `ch:setup` concern.
- **Web dashboard uses a single static `DEV_TOKEN`** (owner of the seeded merchant), so in the live dashboard the switcher only appears once you log in as an agency user. Verified via API with a seeded agency fixture (agency + 2 clients + admin/member + one assignment).
- No unit tests were added for the thin settings/agency HTTP controllers (Zod-parse + service-call + envelope); the underlying services are unit-tested (rbac guard 14, agency access 14, team 10, assignments 6, acting-merchant hook 5) and the guards + switching were verified live end-to-end.
