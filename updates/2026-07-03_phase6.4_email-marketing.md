# Update: Email Marketing (roadmap 6.4)

**Date:** 2026-07-03

**Author:** Claude Code (Session — "LANE 2 — Email Marketing")

## What Was Built

The full email marketing surface behind the frozen `ChannelAdapter` / `message-dispatch`
seam. EMAIL is now a live channel: campaigns (channel EMAIL) and template sends both flow
through Lane A's `message-dispatch` queue/worker into a new EMAIL branch that renders +
delivers per recipient and persists `Message` rows using the freeze-v2 email columns.

**Render engine** (`services/email/render.ts` + `tokens.ts`): drag-drop blocks (text,
image, button, divider, spacer, **dynamic-product**, **conditional-by-segment**) → a
table-based, mobile-safe HTML document + plain-text alternative. Personalization tokens
(`{{customer.first_name|fallback}}`, `{{order.*}}`, `{{merchant.*}}`) resolved and
HTML-escaped (customer-field markup can't inject). Dynamic product blocks resolve **live at
send time** from the Postgres `Product` catalog (`services/email/products.ts`):
`top_sellers` (ranking placeholder until Lane C fills it), `recommended` (Lane D
`Recommendation` cache), `viewed` (ClickHouse `product_view` events, falls back to top
sellers), `manual`. Conditional blocks render only for members of the block's segment
(resolved per recipient in `services/email/context.ts`).

**EmailAdapter** (`lib/channels/email.adapter.ts`): real AWS SES v2 `SendEmail`, SigV4-signed
with native `fetch` (zero new deps — `services/email/sigv4.ts`), Resend fallback when SES
creds are absent, and a clean non-retryable "Email not configured" result when neither is
set (mirrors the WhatsApp adapter; app boots credential-free).

**Dispatch** (`services/email/dispatch.ts` + EMAIL branch in `message-dispatch.worker.ts`):
consent + `EmailSuppression` gate → SKIPPED; per-recipient render (real template or plain
campaign body); Message row created QUEUED (reused across BullMQ retries to avoid duplicate
rows) → SENT/FAILED with the freeze-v2 columns (`toEmail`, `subject`, `bodyHtml`,
`emailTemplateId`, `metadata.abVariantId`); flips the originating `CampaignRecipient`.

**Spam-score** (`services/email/spam-score.ts`): heuristic 0–100 checker (trigger words,
shouty/empty subject, missing unsubscribe, image-heavy body, no text part…) with itemized
issues, surfaced live in the builder.

**Sending domains** (`services/email/domain-verify.ts` + `routes/sending-domains/*`): DKIM
(3 SES-style CNAMEs) + SPF + DMARC record generation and a verify flow — real `node:dns`
lookups when SES creds are present, mocked success when absent (offline-testable).

**A/B testing** (`services/email/ab-test.ts`): deterministic weighted variant assignment on
the frozen `AbTest` table (entityType `EMAIL_TEMPLATE`), subject/blocks override at dispatch,
and a two-proportion z-test winner decision (open_rate / click_rate).

**Public tracking** (`routes/email/tracking.ts`): 1×1 open pixel (stamps `Message.openedAt`)
and HMAC-verified one-click unsubscribe (writes `EmailSuppression`, clears
`Customer.isSubscribedEmail`).

**API** (`routes/email-templates/*`): template CRUD + `preview` + `spam-check` + `test-send`
(synchronous, one address) + `send` (fan-out to a segment through message-dispatch).
Tenant-scoped, Zod-validated, standard response envelope.

**Web** (`routes/email-templates.*`, `routes/sending-domains._index.tsx`,
`components/email/*`): monochrome design-system builder — palette + drag-drop/reorderable
block editor (recursive conditional blocks), autosaving live **desktop/mobile preview**
(server-rendered iframe), live spam score, test-send, and send-to-segment. First feature
pages to adopt `~/components/ui`.

## Decisions Made This Session

- **`MessageDispatchJob` extended additively** with optional `emailTemplateId?` + `abVariantId?`
  (tagged `// lane:email`). An optional field on a frozen interface is backward-compatible —
  WhatsApp/SMS paths ignore it. Needed so template-based sends (dynamic products render at
  send time) can identify the template through the existing queue. **Flag for integrator.**
- **Campaign EMAIL vs template EMAIL both reuse message-dispatch.** Plain campaign bodies
  (Lane B sets only `content.body`) render into a branded HTML shell; rich templates set
  `emailTemplateId`. One EMAIL branch handles both.
- **A/B variant stored on `Message.metadata.abVariantId`** (Message has no dedicated column);
  metrics filter by JSON path.
- **`segmentId`/`customerId` route params are `z.string().min(1)`, not `.cuid()`** — seeded
  rows use readable slugs (`seed-segment-champions`). Caught by the boot smoke test (the
  "cuid id validation" gotcha).
- **Open-tracking pixel needs the Message id before send** → Message row is created QUEUED
  first and reused across retries (no duplicate rows).
- **Two new web files touch design-lane surface** (`nav.ts` leaves, two `icons.tsx` icons):
  additive, tagged. **Flag for integrator.**

## Known Issues Left Open
- **Click-through tracking** not wired (no link rewriting), so `Message.clickedAt` stays
  null → A/B on `click_rate` needs it; `open_rate` works. (Follow-up.)
- **Per-merchant SES rate limiting** not applied on the email path (relies on SES 429 →
  retryable + BullMQ). WhatsApp's app-level limiter is not shared to EMAIL. (Follow-up.)
- **`top_sellers` ranking** is a placeholder (`repurchaseRate90d` desc) until Lane C's
  analytics populate real sales-ranked columns.
- **Per-merchant SES creds** via `MerchantIntegration` not used — adapter reads platform env
  creds only (the `ChannelAdapter.send` signature carries no merchantId). (Follow-up.)
