## What Was Built

The real WhatsApp Cloud API send path behind the frozen `ChannelAdapter` contract
(`packages/shared/src/types.ts`), implementing the approved spec
`docs/superpowers/specs/2026-06-26-whatsapp-channel-adapter-design.md` including all 7 agreed
corrections. SMS and Email remain stubs behind the same interface.

End-to-end flow:
`dispatchChannel()` (signature unchanged) → enqueue `MessageDispatchJob` on the new
`message-dispatch` queue → `message-dispatch.worker` (load customer merchant-scoped → consent gate
→ per-merchant Redis rate limit → resolve template + substitute variables → `adapter.send()` →
persist `Message`). Inbound: `POST /webhooks/whatsapp` (HMAC-verified) applies monotonic status
updates and opt-out; `GET /webhooks/whatsapp` does the Meta verification handshake.

Implemented corrections:
1. **Language** stored/sent as a Meta language-code string (`en`, `ur`, `ar_AE`…), never an enum.
2. **variableMap defaults** — each `{{n}}` entry may carry a `default`; empty-with-no-default is a
   non-retryable local FAILED (`"Empty template variable {{n}}"`) rather than an opaque Meta reject.
3. **Monotonic status** — `QUEUED<SENT<DELIVERED<READ`; status never regresses, FAILED is terminal,
   the matching timestamp is always stamped (a late DELIVERED stamps `deliveredAt` but keeps `READ`).
4. **Opt-out** — STOP/UNSUBSCRIBE text **and** Meta's native marketing opt-out (button reply /
   `marketing_opt_out` marker) both flip `Customer.isSubscribedWhatsapp=false`.
5. **Rate limit** — Redis fixed-window token bucket per merchant; over-cap re-enqueues with a
   deterministic per-job jittered delay (no `Math.random`).
6. **`ChannelSendPayload`** is the frozen channel-tagged discriminated union (WhatsApp/SMS/Email).
7. **Endpoint-selection seam** — `WhatsAppAdapter.resolveEndpoint(category)` isolates the URL so
   MARKETING can later route to MM Lite; version is always `env.META_API_VERSION` (never hardcoded).

## Decisions Made This Session

- **Retryable vs permanent persistence:** permanent failures persist a `Message{status:FAILED}` (audit
  trail for the log UI) + `UnrecoverableError`; retryable failures rethrow a plain `Error` and persist
  NO row, so BullMQ retries without accumulating duplicate FAILED rows across attempts.
- **Stub channels:** the worker skips non-WhatsApp channels with a log and no Message row (spec §4.5
  step 2). SMS/Email adapters exist behind the registry; activating them is a one-line change.
- **Template-variable field allow-list:** `substituteVariables` resolves only an explicit set of
  customer fields (firstName, city, totalSpent, …); unknown fields resolve empty (→ default or FAILED),
  so a template can never read an arbitrary/sensitive column.

## Deviations from Roadmap / Spec

- **Added one env var beyond the spec's two:** `META_WHATSAPP_BUSINESS_ACCOUNT_ID` (optional). Template
  submit-to-Meta is WABA-scoped, and the spec listed only `META_APP_SECRET` + `META_API_VERSION`. Rather
  than half-wire the real submit, this optional var enables the real Graph `message_templates` call when
  present; absent, submit sets `status=PENDING` locally (the offline-testable path the spec requires).

  (declared `^10.3.0` in `apps/api/package.json`) requires Fastify 5, but `fastify` is pinned
  `^4.27.0` (resolves 4.29.1) → `FST_ERR_PLUGIN_VERSION_MISMATCH` at startup. I did NOT touch
  `package.json`/lockfile; this affects `main` and every lane equally. The HTTP health check therefore
  could not complete. Fix (integrator, shared dep): either bump `fastify` to 5.x or pin
  `@fastify/rate-limit` to `^9` (the Fastify-4-compatible major). Code compiles, typechecks, and all
  routes register at build time.
- **Inbound merchant attribution:** with a single app-level WhatsApp number (no per-merchant
  `phone_number_id` in the frozen schema), inbound messages are attributed to the first customer
  matching the `from` phone. Status updates are unaffected (looked up by `wamid`, which carries its own
  `merchantId`). Documented in `routes/webhooks/whatsapp.ts`.
- **SMS/Email** are stubs (later wave: SMS in 6.3 tail, Email in 6.4).
- ClickHouse not run in this lane (WhatsApp messaging is Postgres-only); `ch:setup` skipped.
- Web nav/sidebar links were not added (would touch a shared web file); the new pages cross-link to
