## What Was Built

The inbound side of two-way WhatsApp: non-STOP inbound messages are matched to a `WhatsAppConversation`
(frozen table from schema-freeze-v2) and routed to the waiting context. Two structured contexts plus
free-form:

- **`journey_reply`** — resolves a journey "wait for reply / branch on reply" step. On a matching reply
  the parked enrollment resumes down the branch child (matched by child `label`); on timeout it resumes
  down the timeout branch (or completes).
- **`verification`** — COD-order verification. The reply is classified (confirm/cancel), the conversation
  is resolved+closed, and the decision is handed to the COD verify lane via a documented log **seam**
  (no cross-lane queue/schema coupling — that worker isn't built yet).
- **`freeform`** — an inbound with no active conversation auto-opens an OPEN conversation so the thread
  surfaces in the inbox and the 24h service window is tracked.

**State machine** (`WhatsAppConversationState`): a conversation is born **OPEN** (taking the partial-unique
`(merchant_id, phone) WHERE state='OPEN'` slot), flips to **AWAITING_REPLY** when a reply-expecting prompt
is sent, and settles to **CLOSED** (resolved) or **EXPIRED** (timed out). Reply-vs-timeout is made mutually
exclusive with optimistic `updateMany` guards: `claimStructuredReply` (AWAITING_REPLY→OPEN) and
`expireConversation` (AWAITING_REPLY→EXPIRED, guarded by an exact `awaitingReplyUntil` match so a stale
timeout job can't expire a re-armed wait). Whoever flips the state first wins.

**Awaiting-reply timeout** is a delayed BullMQ job on a new `conversation-timeout` queue (one per await
round, `jobId` keyed on conversation+deadline — BullMQ-idiomatic, exactly like the DELAY step). If the reply
arrives first, the timeout job is a clean no-op.

**Journey executor extension** — the frozen `JourneyStepType` enum has no `WAIT_FOR_REPLY`, so a wait-for-reply
step is an **ACTION whose Json `config` carries a `waitForReply` block** (branch children matched by `label`,
exactly like `CONDITION`'s `true`/`false`). The executor's ACTION case detects this via `isWaitForReplyConfig`,
sends the prompt, opens a `journey_reply` conversation, and **parks** the enrollment (returns without
advancing). Ordinary ACTION steps are completely unaffected.

**Webhook dispatch** — the existing `/webhooks/whatsapp` route now routes non-STOP inbound from a known
customer into the engine (dynamic import, so the queue-backed engine stays out of the Channels-lane webhook
unit tests' static graph). STOP still opt-outs via the existing path and never opens a conversation.

**Inbox UI** — `/conversations` (list with state filter + stat tiles + last-message preview) and
`/conversations/:id` (message thread with inbound/outbound bubbles). Both are the **first** routes in the app
built entirely from the monochrome design system (`~/components/ui`) — state shown by shade/border/icon, never hue.

## Decisions Made This Session

- **Wait-for-reply modeled as ACTION + `config.waitForReply`, not a new step type** — `JourneyStepType` is a
  frozen Prisma enum; adding a value would need a schema change (forbidden in a lane). The Json `config` is the
  correct extension point, and branch children reuse the existing child-by-`label` pattern. Backward-compatible.
- **OPEN is the single routable state; the partial-unique is the determinism backstop.** A conversation is
  always created OPEN (guarded by the unique index; P2002 → re-fetch the winner), then transitions to
  AWAITING_REPLY. `openConversation` always reuses any active (OPEN or AWAITING_REPLY) row.
- **Delayed timeout job (not a periodic sweep)** — precise, idempotent via `jobId`, no polling; mirrors the DELAY step.
- **Verification handoff is a log seam, not a queue enqueue** — the COD verify worker doesn't exist yet;
  enqueuing onto `COD_VERIFICATION` would pile up unconsumed. `classifyVerificationReply` is exported for the COD
  lane to reuse.
- **Auto-open a freeform conversation on an un-matched inbound** — gives the inbox real content and tracks the
  WhatsApp 24h window; only for known customers (tenant-safe: merchantId comes from the resolved customer).
- **Concurrency hardening (from a self-run adversarial review — see below).** (a) `openConversation` and the new
  `reuseOrArmConversation` do find-or-create inside a `$transaction` guarded by a Postgres **advisory lock**
  (`pg_advisory_xact_lock(hashtext(merchant|phone))`) so two concurrent opens for one phone can't create two
  active rows — the partial-unique only covers `state='OPEN'` and a row vacates that slot when it flips to
  AWAITING_REPLY. (b) `reuseOrArmConversation` **detects a conflicting live wait**: if the phone is already
  awaiting a *different* structured context, it refuses to clobber and returns `conflict`, so the later journey
  wait takes its timeout branch deterministically instead of silently stranding the earlier enrollment.
  (c) The `conversation-timeout` queue uses the default retry policy (3 attempts) with a recovery check
  (resume when the row is already EXPIRED-for-this-deadline) and a deterministic dedupe `jobId` on the resume,
  so a transient failure/crash between expire and resume recovers without double-firing the branch.

## Known Issues Left Open

- **One structured wait per phone at a time (by design of the frozen schema).** The schema permits one active
  conversation per (merchant, phone). If the same customer is simultaneously enrolled in two wait-for-reply
  journeys, only the first holds the conversation; the second is **deterministically short-circuited to its
  timeout branch** (no prompt sent, no silent strand — this is the hardening above, verified by unit test).
  A future schema change (per-context conversations) would allow genuinely concurrent waits per phone.
- **Inbound reply resolution is at-most-once.** The webhook always returns 200, so a crash mid-`dispatchInbound`
  (after persisting the Message but before resolving the wait) is not retried. This matches the existing webhook
  path's semantics; the *timeout* path (which has a retry queue) is crash-recoverable. A durable inbound queue
  would close this, but that touches Channels-lane webhook internals (out of this lane's scope).
- **COD verify handoff is a log seam** — the COD lane must replace `resolveVerificationReply` / `verificationTimeout`
  logs with a real consumer (read the classified reply off the inbound Message + `contextId`).
- **`listConversations` last-message preview is a per-row query (N+1)** — fine at inbox page sizes (≤50);
  revisit if volume grows.
